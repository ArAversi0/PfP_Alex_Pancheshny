# 02 — Требования и предметное моделирование

## Обзор этапа

На данном этапе выполнен переход от бизнес-модели (этап 01) к формализованным системным требованиям. Основной целью этапа является описание того, **как система должна реализовывать бизнес-логику**, зафиксированную ранее.

В рамках этапа были сформированы:

- диаграмма вариантов использования;
- модель предметной области;
- спецификации ключевых пользовательских сценариев;
- таблица трассировки требований.

Этап закладывает основу для дальнейшего проектирования архитектуры и реализации системы.

---

## Цели этапа

Основные цели этапа:

- трансформация бизнес-требований в системные;
- формализация взаимодействия пользователей с системой;
- определение структуры доменной модели;
- фиксация бизнес-правил на уровне системы;
- подготовка базы для проектирования backend и базы данных.

---

## Модель вариантов использования

### Акторы системы

В системе выделены следующие акторы:

- **ROLE_GUEST** — неавторизованный пользователь desktop-клиента (offline режим);
- **ROLE_USER** — основной пользователь системы;
- **ROLE_ADMIN** — администратор;
- **OAuth2 Provider** — внешняя система аутентификации.

---

### Ключевые варианты использования

Основные сценарии системы:

- регистрация и аутентификация;
- создание и редактирование персонажа;
- управление инвентарём;
- экипировка предметов;
- бросок виртуальных костей;
- импорт и экспорт персонажа;
- работа со справочниками;
- управление пользователями и контентом (администратор).

---

### Особенности модели

- использование связей:
  - `<<include>>` — для обязательной бизнес-логики (пересчёт характеристик, overweight);
  - `<<extend>>` — для расширяемых сценариев (OAuth2);
- чёткое разделение ролей;
- отсутствие realtime и сложных игровых механик (в соответствии с ограничениями проекта).

---

## Предметная модель

### Общая структура

Предметная модель построена в соответствии с принципами предметно-ориентированного проектирования и включает:

- Entity;
- Value Object;
- Enum;
- связи (composition, association, inheritance).

---

### Ключевые агрегаты

#### Character (Aggregate Root)

Центральная сущность системы, объединяющая:

- CharacterStats;
- CharacterCondition;
- Inventory.

Отвечает за:
- расчёт характеристик;
- применение эффектов;
- целостность данных персонажа.

---

### Основные сущности

- **User** — пользователь системы;
- **Role** — роль пользователя;
- **Inventory** — инвентарь;
- **InventorySlot** — слот инвентаря;
- **Item** — базовый предмет;
- **Equipment** — экипируемый предмет;
- **CharacterCondition** — состояние персонажа;
- **Spell** — заклинание.

---

### Value Objects

- **CharacterStats** — характеристики персонажа;
- **SpellDescription** — описание заклинания.

---

### Enum

- EquipmentType;
- SpellType.

---

### Ключевые бизнес-правила

- лимит персонажей зависит от подписки;
- overweight влияет на скорость;
- урон считается по частям тела;
- offline режим доступен только для desktop guest;
- характеристики пересчитываются автоматически.

---

Были детализированы ключевые сценарии:

### UC-003 — Создание персонажа

## Трассировка требований

На этапе сформирована связь между:

- бизнес-требованиями (этап 01);
- вариант использования;
- предметная модель;
- системными функциями.

Это обеспечивает:

- прозрачность требований;
- контроль покрытия функциональности;
- основу для тестирования.

## Ограничения и допущения

- система не является realtime;
- отсутствует rules engine;
- offline поддержка реализуется только для desktop;
- проект остаётся modular monolith;
- не вводятся микросервисы.

---

## Итог

Этап 02 фиксирует систему как **формализованный инженерный объект**, готовый к дальнейшему архитектурному проектированию и реализации.

Основной результат — согласованная связь:

**Бизнес-требования → варианты использования → предметная модель → системные требования**
