# Руководство администратора

Админский контур PfP Companion System доступен в web-версии приложения. Desktop-клиент не содержит админки: он предназначен для игроков и работает с персонажами, дополнительным контентом и настройками.

## Назначение админки

Админка используется для:

- просмотра общей статистики системы;
- управления пользователями;
- просмотра персонажей всех пользователей в режиме только для чтения;
- редактирования лора и справочника правил.

Админские функции не должны использоваться для обычной игры. Они нужны для сопровождения контента, контроля пользовательских профилей и проверки данных.

## Получение доступа

Пользователь должен иметь роль администратора в базе данных. На текущем MVP-этапе назначение администратора выполняется вручную через обновление записи пользователя в БД.

Типовой порядок:

1. Пользователь регистрируется или входит через Google.
2. В таблице пользователей находится его email.
3. Роль пользователя обновляется на admin-роль.
4. Пользователь заново входит в систему.
5. В верхнем меню появляется пункт `Admin`.

Для промышленной версии рекомендуется добавить отдельный безопасный сценарий назначения администраторов: начальный администратор, консольная команда или закрытый endpoint только для владельца системы.

## Панель администратора

Страница `/admin` показывает три основных блока:

- `Users` - количество пользователей и переход к управлению пользователями.
- `Characters` - количество персонажей и переход к просмотру всех листов.
- `Content` - количество материалов лора/справочника и переход к редактированию контента.

Все блоки должны иметь единый стиль: тёмная карточка, золотой контур, заголовок, числовой показатель и кнопка действия.

## Управление пользователями

Страница `/admin/users` предназначена для просмотра пользователей и административных действий над профилями.

Администратор видит:

- email пользователя;
- роль;
- источник входа;
- статус подтверждения email;
- дату создания;
- доступные действия.

Если удаление профиля включено, оно должно использоваться осторожно. Удаление пользователя обычно затрагивает связанные персонажи и данные аккаунта. Перед production-использованием желательно добавить подтверждение действия и аудит.

## Просмотр персонажей пользователей

Страница `/admin/characters` показывает персонажей всех пользователей в режиме только для чтения.

Требования к отображению:

- персонажи группируются по email владельца;
- у каждого пользователя показывается строка вида `N of 100 slots used`;
- под строкой отображаются карточки персонажей этого пользователя;
- между пользователями есть визуальное разделение;
- администратор может открыть лист персонажа только для просмотра;
- редактирование листа в этом режиме недоступно.

На странице предусмотрены пагинация по пользователям и поиск по email. Поиск запускается по кнопке, чтобы страница не прыгала во время ввода.

## Редактирование контента

Страница `/admin/content` используется для редактирования Lore и Rule book.

Контент хранится как статьи и категории. Категория тоже может открываться как статья, даже если её наполнение пока пустое. Для текста используется Markdown.

Администратор может:

- выбрать раздел Lore или Rule book;
- открыть категорию или статью;
- изменить markdown-содержимое;
- сохранить изменения;
- проверить отображение статьи в пользовательском интерфейсе.

В MVP не реализуется синхронизация этого контента с desktop-клиентом. Desktop-приложение использует встроенные JSON-версии лора и справочника в режиме только для чтения.

## Рекомендации по эксплуатации

- Не хранить реальные секреты OAuth2 и SMTP в репозитории.
- Использовать Mailpit только для локальной разработки.
- Для реальной почты подключать внешний SMTP-провайдер.
- Перед удалением пользователей делать резервную копию БД.
- После изменения справочника проверять markdown-рендеринг на web-странице.
- Для изменения desktop-контента выпускать новую сборку приложения или отдельно реализовывать механизм обновления контента.

## Ограничения MVP

- Администратор назначается вручную через БД.
- Нет журнала аудита административных действий.
- Нет отдельной роли owner/superadmin.
- Нет промышленного почтового провайдера по умолчанию.
- Desktop-клиент не имеет админских функций.
