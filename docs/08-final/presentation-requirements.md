# Требования к презентации проекта

Цель презентации - наглядно показать PfP Companion System как enterprise web + desktop приложение для собственной НРИ-системы PfP.

## Формат

- Формат файла: `.pptx`.
- Рекомендуемый объём: 12-15 слайдов.
- Язык: русский.
- Стиль: тёмный фон, золотые акценты, крупные заголовки, минимум лишнего текста.
- На каждом слайде должен быть один основной смысл.

## Структура презентации

### 1. Титульный слайд

Содержит:

- название проекта: PfP Companion System;
- расшифровку PfP: Pain for Pleasure;
- тип проекта: enterprise-приложение web + desktop;
- ФИО студента и учебную группу.

### 2. Проблема и цель

Показать, зачем нужен проект:

- хранение листов персонажей;
- синхронизация web и desktop;
- единый справочник правил и лора;
- поддержка собственной игровой системы.

### 3. Пользователи и роли

Отразить:

- гость desktop-приложения;
- пользователь аккаунта;
- администратор.

Лучше использовать простую схему ролей.

### 4. Архитектура

Показать схему:

```text
Web Client -> Backend API -> PostgreSQL
Desktop-режим аккаунта -> Backend API -> PostgreSQL
Desktop-гостевой режим -> Local JSON
Web-админка -> Backend API -> управление контентом, пользователями и персонажами
```

### 5. Backend

Показать стек и модули:

- Spring Boot;
- JPA;
- PostgreSQL;
- Flyway;
- Security/JWT/OAuth2;
- Mailpit/SMTP;
- модули identityaccess, charactersheet, content, charactertransfer.

### 6. Web-клиент

Показать скриншоты:

- главная страница;
- архив персонажей;
- лист персонажа.

Кратко перечислить функции: CRUD, JSON import/export, inventory, spells, lore/rule book.

### 7. Desktop-клиент

Показать скриншоты:

- главное меню;
- гостевой архив и архив аккаунта;
- desktop character sheet.

Отдельно подчеркнуть:

- гостевой режим без backend;
- режим аккаунта с синхронизацией;
- устанавливаемый app image.

### 8. Лист персонажа

Показать web или desktop sheet и отметить блоки:

- Character Info;
- Stats and Skills;
- Condition;
- Equipment;
- Inventory;
- Spells;
- Additional Info.

### 9. Инвентарь и заклинания

Показать:

- ячейки инвентаря;
- drag-and-drop;
- создание предмета из пустой ячейки;
- экипировку;
- карточку заклинания.

### 10. Lore и Rule book

Показать:

- страницу справочника;
- дерево категорий;
- статью;
- markdown-рендеринг.

Уточнить, что в desktop MVP контент встроенный и доступен только для чтения.

### 11. Админка

Показать:

- панель администратора;
- пользователи;
- персонажи, сгруппированные по email;
- редактор контента.

### 12. Тестирование

Показать:

- успешный запуск backend-тестов;
- JaCoCo HTML report;
- покрытие 63%;
- desktop tests;
- web build.

### 13. Сборка и установка

Показать:

- Gradle task `packageDesktopAppImage`;
- папку `PfP Companion`;
- запуск `PfP Companion.exe`;
- примечание про WiX для installer.

### 14. Статистика разработки

Вставить скриншоты:

- GitHub Insights Commit Activity;
- GitHub Insights Punch Card.

Если GitHub ещё не подключён, этот слайд можно временно оставить как placeholder до загрузки репозитория.

### 15. Итоги и дальнейшее развитие

Перечислить:

- проект доведён до MVP+;
- реализованы web, desktop, backend, admin, sync;
- дальнейшие шаги: production SMTP, удобное назначение admin, CI/CD, подписанный installer, обновление desktop-контента.

## Обязательные демонстрационные скриншоты

- Web main page.
- Web character archive.
- Web character sheet.
- Desktop main menu.
- Desktop account archive.
- Desktop character sheet.
- Lore или Rule book.
- Панель администратора.
- JaCoCo coverage report.
- Desktop packaged app.

## Демонстрационный сценарий

Для устной защиты удобно показать такой порядок:

1. Запустить backend и web.
2. Открыть web archive.
3. Открыть персонажа и показать лист.
4. Перейти в desktop-режим аккаунта.
5. Показать, что тот же персонаж доступен в desktop.
6. Изменить небольшое поле и сохранить.
7. Показать JSON export.
8. Открыть Lore/Rule book.
9. Кратко показать admin dashboard.

## Что не стоит перегружать

- Не показывать слишком много кода на слайдах.
- Не вставлять длинные JSON-файлы.
- Не показывать реальные секреты, токены и пароли.
- Не смешивать web и desktop скриншоты на одном слайде, если они становятся мелкими.
