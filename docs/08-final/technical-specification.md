# Техническое задание

## 1. Общие сведения

Наименование проекта: **PfP Companion System**.

Тип проекта: enterprise web + desktop приложение для сопровождения собственной настольной ролевой системы **PfP (Pain for Pleasure)**.

Назначение: предоставить игрокам и мастеру единое приложение для создания, хранения, редактирования, импорта, экспорта и синхронизации листов персонажей, а также чтения лора и справочника правил.

## 2. Пользователи и роли

### Гость desktop-приложения

Возможности:

- создавать локальные листы персонажей;
- хранить до 30 локальных персонажей;
- импортировать и экспортировать JSON;
- редактировать лист персонажа;
- использовать Lore и Rule book в режиме только для чтения;
- использовать настройки desktop-приложения.

Ограничения:

- нет синхронизации с backend;
- нет доступа к веб-аккаунту;
- нет админских функций.

### Зарегистрированный пользователь

Возможности:

- регистрироваться и входить по email/паролю;
- подтверждать email;
- входить через Google OAuth2;
- создавать до 100 персонажей;
- редактировать персонажей в веб-версии и desktop-режиме аккаунта;
- синхронизировать персонажей между веб-версией и desktop;
- импортировать и экспортировать JSON;
- читать Lore и Rule book.

### Администратор

Возможности:

- входить в панель администратора;
- управлять пользователями;
- просматривать персонажей всех пользователей в режиме только для чтения;
- редактировать Lore и Rule book в web-админке.

Ограничения:

- админский режим доступен только в веб-версии;
- desktop-клиент админку не содержит.

## 3. Функциональные требования

### Авторизация

- Система должна поддерживать регистрацию по email и паролю.
- Система должна отправлять письмо подтверждения email через SMTP-адаптер.
- В development-режиме письма должны перехватываться Mailpit.
- Система должна поддерживать вход через Google OAuth2.
- После входа пользователь получает JWT-токен.
- Web-клиент должен автоматически прикладывать JWT к защищённым запросам.
- При 401 пользователь должен получать понятное сообщение или сценарий выхода, а не технический stack trace.

### Персонажи

- Пользователь может создать персонажа.
- После создания персонажа пользователь должен попадать на лист персонажа.
- Пользователь может открыть, изменить, удалить персонажа.
- Количество персонажей account-пользователя ограничено 100.
- Desktop-гостевой режим ограничен 30 персонажами.
- Персонаж может быть импортирован из JSON.
- Любой персонаж может быть экспортирован в JSON.

### Лист персонажа

Лист должен содержать:

- портрет;
- имя, уровень, происхождение, предысторию, класс, специализацию;
- Defence, Dodge, Blessings, Inspirations, Movement, Health, Money;
- здоровье по частям тела;
- Stats and Skills;
- Equipment;
- Inventory;
- Spells;
- Additional Info.

Игровая логика должна соответствовать системе PfP, а не DnD/Pathfinder.

### Инвентарь

- Инвентарь отображается сеткой по 10 слотов в строке.
- Пустая ячейка подсвечивается при наведении.
- Клик по пустой ячейке открывает создание предмета именно в этой ячейке.
- Предметы можно перемещать drag-and-drop.
- Предметы можно редактировать после создания.
- Предметы подходящего типа можно экипировать.
- Экипированные предметы можно снять.
- Trade-предметы можно продавать.
- Изображение предмета должно помещаться в ячейку без изменения размера сетки.

### Заклинания

- Пользователь может создавать, редактировать и удалять заклинания.
- Карточка заклинания должна поддерживать изображение.
- Окно создания заклинания должно быть стилизовано в общем стиле приложения.

### Лор и справочник правил

- Веб-клиент получает контент из backend.
- Администратор может редактировать контент в web-админке.
- Desktop-клиент для MVP использует встроенный контент в режиме только для чтения.
- Категория может открываться как статья.
- Markdown должен отображаться корректно, включая списки и нумерацию.

### Настройки desktop-приложения

Desktop-приложение должно поддерживать:

- выбор разрешения окна до 1920x1080;
- оконный и полноэкранный режим;
- переназначение горячих клавиш;
- музыкальные темы;
- громкость музыки;
- громкость звуковых эффектов.

## 4. Нефункциональные требования

- Интерфейс должен быть читаемым на desktop-разрешениях от 1024x640 до 1920x1080.
- Web и desktop должны сохранять единый визуальный стиль.
- Сохранение листа не должно вызывать заметных лишних перезагрузок UI.
- Ошибки должны быть человекочитаемыми.
- Локальное guest-хранилище desktop должно работать без backend.
- Desktop-режим аккаунта должен корректно сообщать о недоступности backend.
- Desktop app image должен запускаться без установленной Java.

## 5. Архитектура

Backend:

- Spring Boot REST API.
- PostgreSQL.
- Flyway migrations.
- Spring Security.
- JWT.
- OAuth2 Client.
- Spring Data JPA.
- SMTP email adapter.

Web:

- React.
- TypeScript.
- Vite.
- React Router.
- Axios.
- Feature-based structure.

Desktop:

- JavaFX.
- API-клиент для режима аккаунта.
- JSON-хранилище для гостевого режима.
- Встроенные ресурсы дополнительного контента.
- Settings store.
- Audio subsystem.
- jpackage build.

## 6. Тестирование

Обязательные проверки:

- backend unit/integration tests;
- desktop tests;
- web production build;
- JaCoCo coverage не ниже 40%;
- ручной сценарий регистрации и подтверждения email;
- ручной сценарий Google OAuth2;
- ручной сценарий web/desktop sync;
- ручной сценарий JSON import/export;
- ручной запуск packaged desktop app image.

Актуальное backend-покрытие JaCoCo: **63% instruction coverage**.

## 7. Ограничения MVP

- Production SMTP не подключён по умолчанию.
- Администратор назначается вручную через БД.
- Desktop не синхронизирует Lore и Rule book с backend.
- Desktop installer требует WiX Toolset; без WiX используется app image.
- CI/CD и production deployment не входят в текущий MVP.
- Ограничение частоты запросов для публичных endpoint-ов запланировано как следующий шаг после MVP.
