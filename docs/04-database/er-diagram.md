# ER-диаграмма базы данных

## 1. Общая характеристика

На данном этапе проекта была спроектирована реляционная модель базы данных системы сопровождения персонажей настольно-ролевой системы PFP.

ER-диаграмма отражает структуру хранения данных, соответствующую ранее разработанной предметной модели (ЛР2) и архитектуре PCMEF (ЛР3).

Основные цели проектирования БД:

- обеспечение целостного хранения игровых данных;
- поддержка бизнес-логики системы;
- нормализация структуры данных;
- обеспечение расширяемости;
- соответствие enterprise-подходу.

## 2. Основные сущности

В базе данных были выделены следующие ключевые группы сущностей:

### 2.1. Пользователи

- **users** — хранение учетных записей пользователей;
- содержит:
  - email;
  - password_hash;
  - role;
  - created_at.

Роль хранится как ограниченное множество значений (ROLE_GUEST, ROLE_USER, ROLE_ADMIN).

---

### 2.2. Персонажи

- **characters** — основная сущность персонажа;
- связь:
  - один пользователь → много персонажей.

---

### 2.3. Характеристики персонажа

- **character_stats** — числовые характеристики;
- связь:
  - один к одному с character;
- содержит:
  - strength, dexterity, stamina, intelligence и др.

---

### 2.4. Состояние персонажа

- **character_condition** — система здоровья;
- включает:
  - общее HP;
  - HP отдельных частей тела (head, torso, arms, legs).

Особенность:
- поддержка локального урона.

---

### 2.5. Инвентарь

- **inventory** — контейнер предметов;
- **inventory_slots** — ячейки инвентаря;
- связь:
  - один персонаж → один инвентарь;
  - один инвентарь → множество слотов.

---

### 2.6. Предметы

- **items** — базовая сущность предмета;
- **equipment** — расширение предмета (наследование);
- содержит:
  - тип экипировки;
  - параметры влияния на характеристики.

---

### 2.7. Магия

- **spells** — игровые заклинания;
- **spell_descriptions** — описание заклинаний;

Связь:
- один к одному (value object).

---

### 2.8. Связи между сущностями

Ключевые связи:

- users 1 → N characters  
- characters 1 → 1 character_stats  
- characters 1 → 1 character_condition  
- characters 1 → 1 inventory  
- inventory 1 → N inventory_slots  
- inventory_slots 0..1 → item  
- item → equipment (наследование)  
- spells 1 → 1 spell_descriptions  

---

## 3. Особенности модели данных

В процессе проектирования были учтены следующие особенности:

- разделение Entity и Value Object (перенос в таблицы);
- нормализация структуры (исключение дублирования);
- использование внешних ключей для целостности;
- поддержка бизнес-правил:
  - overweight;
  - локальный урон;
  - ограничения ролей;
- подготовка к масштабированию.

---

## 4. Соответствие архитектуре PCMEF

Разработанная ER-модель соответствует слою **Foundation** архитектуры:

- таблицы → persistence representation;
- репозитории → доступ к данным;
- связи → отражение предметной модели.

Таким образом, БД:

- полностью изолирована от Presentation;
- используется через Repository слой;
- соответствует принципу направленных зависимостей.

---

## 5. Заключение

Спроектированная ER-диаграмма:

- отражает предметную область системы;
- согласована с предметной моделью;
- обеспечивает основу для реализации backend-части;
- соответствует enterprise-подходу к проектированию данных.
