# 04 — Проектирование базы данных

## 1. Общая информация

На данном этапе была выполнена разработка модели базы данных системы сопровождения персонажей настольно-ролевой системы PFP.

Этап является продолжением:

- ЛР2 — предметная модель;
- ЛР3 — архитектура PCMEF.

Основной результат этапа — формализация слоя хранения данных (Foundation).

---

## 2. Цели этапа

- преобразование предметной модели в реляционную модель;
- проектирование структуры базы данных;
- определение сущностей и связей;
- подготовка DDL-описания;
- обеспечение соответствия архитектуре PCMEF.

## 4. Архитектурный контекст

Данный этап соответствует слою:

**Foundation Layer (PCMEF)**

В рамках архитектуры:

- Entity → отображаются в таблицы;
- Repository → работает с БД;
- Service → использует репозитории;
- Controller → не имеет прямого доступа к БД.

## 6. Особенности реализации

- PostgreSQL как основная СУБД;
- использование BIGSERIAL / BIGINT;
- каскадное удаление (ON DELETE CASCADE);
- строгие ограничения (CHECK, NOT NULL);
- разделение логики данных и бизнес-логики.

---

## 7. ORM-стратегия

В backend используется Spring Data JPA. Entity-слой описывает доменные объекты, а Foundation-слой отвечает за сохранение и загрузку данных через репозитории и адаптеры. Миграции схемы выполняются через Flyway, поэтому структура БД фиксируется не только JPA-аннотациями, но и SQL-миграциями.

| Доменная сущность | Таблица | Назначение | Особенности ORM |
| --- | --- | --- | --- |
| `User` | `users` | Профиль пользователя, email, роли, источник входа | Уникальный email, связь с refresh/email/password token таблицами |
| `Character` | `characters` | Основной лист персонажа | Владелец через user id, публичный UUID, каскадные связи с деталями листа |
| `CharacterStats` | `character_stats` | Характеристики и навыки персонажа | Связь один-к-одному с персонажем, значения используются для расчётов |
| `CharacterCondition` | `character_conditions` | Здоровье по частям тела | Связь один-к-одному, хранение текущих и максимальных значений |
| `Inventory` | `inventories` | Инвентарь персонажа | Один инвентарь на персонажа |
| `InventorySlot` | `inventory_slots` | Ячейки инвентаря | Позиция слота, связь с предметом, поддержка drag-and-drop порядка |
| `Item` | `items` | Предметы, экипировка, trade-предметы | Тип предмета определяет допустимые поля и действия |
| `Spell` | `spells` | Заклинания персонажа | Связь с персонажем, текстовые поля и URL изображения |
| `ContentNode` | `content_nodes` | Lore и Rule book | Иерархия категорий и статей, markdown-содержимое |

Основные правила маппинга:

- бизнес-идентификаторы доступны клиентам как UUID, внутренние технические идентификаторы остаются на стороне БД;
- связи с вложенными частями листа сохраняются каскадно там, где объект не имеет смысла без персонажа;
- коллекции инвентаря и заклинаний обновляются через прикладные сервисы, а не напрямую из контроллеров;
- SQL-ограничения и JPA-валидация дополняют друг друга;
- миграции Flyway являются источником истины для структуры production-БД.
